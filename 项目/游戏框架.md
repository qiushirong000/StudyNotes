### 消息框架：
- 消息类：消息的类型，命令和参数
- 消息中心类：逐渐下发消息给管理类，
- 管理类：如UI管理类、声音管理类、角色管理类等。接收消息中心类的消息，向下下发消息给功能脚本。
- 功能脚本：如血量脚本、背包脚本、地图脚本和菜单脚本。通过消息中心类发送消息。

### 管理器：
- 资源管理类：资源管理器制定了一个资源存放的规则，包含资源命名、资源存储路径、路径命名等。提供统一的资源动态加载的接口。包括：（1）资源类：资源对象，资源类型，路径，读取次数。（2）资源管理器：已经加载的资源字典，加载并生成对象（同步、协程、异步），获取资源。

- UI管理类：三层，UIManager、UIContainer、UIPanel；UIManager中通过字典存储总的UIContainer，通过UIManager的单例进行各个控件的相互调用。

- UI管理器2：UI_Type（存储单个UI的信息，包括名字和路径）、BasePanel（所有UI面板的父类，包括UI面板的状态信息）、UIManager（单例，存储所有UI信息、创建或销毁UI）、PanelManager（面板管理器，用栈管理当前显示的UI）。

### 日志系统：
核心分层是：ELogger 枚举定义模块分组（Logger.cs）→ Log 静态门面路由（Log.cs）→ LogBus ScriptableObject 承载每个模块的开关、等级、着色等配置（LogBus.cs）→ 最终调用 Unity 的 Debug.Log/LogWarning/LogError。

启动时通过 [AppInitialize] 从 YooAsset 批量加载所有 LogBus 资产，按 ELogger 注册到字典。业务侧不直接调 Log.Error(ELogger.World, ...)，而是通过 ModuleLogger 子类（如 WorldLogger.Info(...)、D.Error(...)）写日志，编译期绑定模块，调用更简洁。

所有日志 API 都标了 [Conditional("UNITY_EDITOR")] 和 [Conditional("LOG_VERBOSE")]，正式包默认整段日志代码被编译器剔除，不影响性能。编辑器里可在 Inspector 改每个 LogBus 的 enabled、LogLevel（Flags 位掩码）、颜色、filter 标签，实现分模块、分等级过滤。

另外有 LogCollect 把运行时上下文采集到 CrashSight，CrashSightLog 对接崩溃上报；这与 LogHelper（DEBUG_V 条件编译的通用日志）是并行体系，面向线上诊断。


Logger.cs：定义 ELogger 模块分组枚举（文件名是 Logger，内容是枚举）
Log.cs：静态门面 + 初始化 + CrashSightLog
LogBus.cs：单模块日志总线（SO 配置 + 过滤 + 着色 + 输出）
ModuleLogger.cs：各模块 Logger 子类 + 泛型基类
LogLevel.cs：日志等级 Flags 枚举
LoggerExtension.cs：ELogger 扩展方法
LogCollect.cs：运行时上下文采集上传 CrashSight

### 状态机：
- 状态基类FSMState，属性包括状态StateId，状态拥有者Mono，状态所属管理器FSMManger；方法包括进入状态方法OnEnter()，在状态中每帧调用的方法OnUpdate()。
- 状态管理类FSMManager，属性包括状态管理列表List<FSMState> StateList，当前状态CurrentIndex；方法包括改变状态ChangeState（int StateID），Update方法。
- 状态类：重写状态基类方法，在OnEnter中执行对应状态的动画，在Update中监听是否要变为其他状态。
